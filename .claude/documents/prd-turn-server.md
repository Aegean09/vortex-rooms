# PRD: TURN Server Support

**Status:** Approved — ready for implementation
**Version:** 0.1
**Author:** Product Owner
**Date:** 2026-03-07
**Related issues:** VR-roadmap (Priority #2)

---

## Problem / Goal

Vortex currently uses only Google STUN servers for WebRTC NAT traversal. STUN is sufficient when both peers are on typical home networks, but fails when either peer is behind a strict corporate firewall, symmetric NAT, or certain mobile carrier networks. When STUN fails, the peer connection silently never establishes — users see the other person in the participant list but hear no audio.

Adding TURN (Traversal Using Relays around NAT) ensures audio connects on nearly any network by relaying media through a server as a fallback.

---

## User Stories

- As a user on a corporate network, I want voice to connect reliably, so I don't have to switch to my phone hotspot to use Vortex.
- As a user reporting "audio doesn't work", I want the app to handle strict NAT automatically, so I don't need to troubleshoot network settings.
- As the product team, we want to reduce the "voice never connects" support category to near zero.

---

## Acceptance Criteria

- [ ] Voice connects successfully between two peers where at least one is behind symmetric NAT (verified via test)
- [ ] TURN credentials are never exposed in client-side source code or network responses beyond the ICE negotiation
- [ ] TURN is used only as a fallback — direct STUN connection is preferred when possible (no added latency for peers that work today)
- [ ] ICE server config is loaded from environment variables, not hardcoded
- [ ] No regression in connection success rate for peers that currently work with STUN only
- [ ] Works in both browser and Tauri desktop

---

## Technical Approach

### ICE server configuration

Update `ICE_SERVERS` in `src/lib/webrtc/webrtc.ts`:

```typescript
const ICE_SERVERS = {
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    { urls: 'stun:stun1.l.google.com:19302' },
    {
      urls: process.env.NEXT_PUBLIC_TURN_URL,
      username: process.env.NEXT_PUBLIC_TURN_USERNAME,
      credential: process.env.NEXT_PUBLIC_TURN_CREDENTIAL,
    },
  ],
};
```

STUN remains first — browser tries direct connection before falling back to TURN.

### TURN server options

**Option A — Self-hosted coturn** (recommended for cost control)
- Run coturn on a small VPS (Hetzner ~€5/mo, DigitalOcean ~$12/mo)
- Open UDP port range (49152–65535) and TCP 443 for fallback
- Use time-limited TURN credentials generated by a Cloud Function (reduces abuse)
- ~$5–20/month

**Option B — Managed TURN (Xirsys, Metered)**
- No server to maintain
- Pay per GB relayed
- ~$0–50/month depending on usage

**Option C — LiveKit/Daily (bundled TURN)**
- If SFU migration happens first, TURN is included
- Defers this work but delays SFU migration

**Recommendation:** Start with a managed TURN provider (Xirsys free tier or Metered) to validate the fix quickly. Move to self-hosted coturn if costs become significant.

### Credential security

Static TURN credentials in `NEXT_PUBLIC_*` env vars are acceptable for v1 — the credentials are only usable for TURN relay, not for any privileged access. For higher security, a Cloud Function can generate short-lived credentials (HMAC-based), but this adds complexity.

### Environment variables

```
NEXT_PUBLIC_TURN_URL=turn:your-turn-server.com:3478
NEXT_PUBLIC_TURN_USERNAME=vortex
NEXT_PUBLIC_TURN_CREDENTIAL=your-credential
```

Add to `.env.example` and App Hosting secrets.

### Graceful degradation

If TURN env vars are not set, the ICE server list falls back to STUN only — existing behavior preserved. Filter undefined entries before passing to `RTCPeerConnection`.

## Out of Scope

- Self-hosted coturn setup and configuration (infrastructure, not code)
- Time-limited credential rotation via Cloud Function (v2 enhancement)
- TURN for screen share video (audio TURN is the priority; video will also benefit)
- SFU architecture (separate PRD)

---

## Success Metrics

- Reduction in user-reported "audio doesn't connect" issues
- Test: peer connection established between two machines on networks known to fail STUN-only

---

## Dependencies and Risks

- **Dependency:** A TURN server must be provisioned before this ships (infrastructure task)
- **Risk:** TURN relays all media through the server — bandwidth cost. With 2 relayed users at 50kbps audio = ~36MB/hour. Negligible for small rooms; monitor if scale increases.
- **Risk:** TURN credentials in `NEXT_PUBLIC_*` are visible in the browser. Acceptable for TURN relay (no privileged access), but document this clearly.
