# QA Test Plans — VR-32, VR-33, VR-34, VR-35

---

## Code Review Findings

1. **VR-34: Dead code** — `optimizeAudioSenderParams()` is defined in `peer-connection-service.ts` but never called from any call site. The SDP munging in `webrtc.ts` handles bitrate capping via `maxaveragebitrate=32000`, so the omission is not a bug, but the function is dead code.

2. **VR-34: Unused config flag** — `DISABLE_NATIVE_NS_WITH_RNNOISE` is exported from `app-config.ts` but the behavior in `audio-helpers.ts` is hardcoded (always disables native NS when RNNoise is on) instead of being gated by this flag.

3. **VR-32: Cold start race** — The 500ms `sleep` in the cold-start deep-link handler is a fixed heuristic. On slow machines the WebView might not have loaded yet; `window.location.href` could run against the wrong origin.

4. **VR-32: Browser handoff edge case** — `tryDesktopDeepLink()` fires both an iframe and `window.location.href` (100ms apart). On browsers that don't handle iframe deep-link dispatch (e.g., Safari mobile), `window.location.href` could navigate the browser away before the 1.5s timeout fires.

## VR-32 — Tauri Deep Linking for Invite-Only Rooms

### Repro Steps

**Scenario A — App already running, user clicks invite link in browser**
1. Build and install the Tauri desktop app (`npm run tauri:build`).
2. Launch the app. Create an invite-only room. Copy the invite link.
3. Open the invite link in a browser (Chrome or Edge).
4. Expected: Browser briefly shows "Opening Vortex...", desktop app focuses and navigates to the invite route.

**Scenario B — App not running (cold start)**
1. Ensure Vortex desktop app is closed.
2. Open the invite link in a browser. Wait up to 2s.
3. Expected: OS launches the Tauri app, navigates to the invite route after ~500ms startup delay.

**Scenario C — App not installed**
1. On a machine without the Vortex desktop app, open the invite link in a browser.
2. Expected: After 1.5s the web flow proceeds (anonymous auth → redirect to setup).

**Scenario D — Direct vortex:// URI click**
1. With desktop app installed, paste `vortex://session/{id}/invite/{token}` in a browser or trigger from another app.
2. Expected: OS opens the Vortex app and navigates to the invite route.

**Scenario E — Second-instance protection**
1. App is running. Trigger a second launch via the deep-link handler.
2. Expected: Existing window is focused, navigates to correct page. No duplicate windows.

**Scenario F — Inside Tauri WebView**
1. Navigate to an invite URL within the desktop app directly.
2. Expected: Deep-link attempt is skipped (`isTauri()` short-circuits), web flow proceeds.

### Edge Cases

- **Malformed deep-link URL:** `vortex://session//invite/TOKEN`, `vortex://session/ABC/invite/`, `vortex://other/route` — `parse_deep_link_path` returns `None`; no navigation; no panic.
- **Wrong keyword:** `vortex://session/ABC/join/TOKEN` — `segments[2]` is `"join"` not `"invite"`, returns `None`.
- **Window minimized:** App running but minimized — `unminimize()` + `show()` + `set_focus()` called before `eval()`.
- **JS injection safety:** `focus_and_navigate` escapes single quotes. Test tokens with special characters.
- **React strict mode:** `deepLinkTriedRef.current` guard prevents double `tryDesktopDeepLink()` in dev mode.

### Regression Checklist

- [ ] Web join flow (non-Tauri) for invite-only rooms still works end-to-end
- [ ] Web join flow for public rooms unaffected
- [ ] Tauri app builds without errors after Cargo.toml changes
- [ ] `capabilities/default.json` has `deep-link:default` permission, no other permissions removed
- [ ] Single-instance plugin still works without a deep link
- [ ] Tray icon menu (Show/Hide/Quit) still functions
- [ ] Cold-start updater check still fires
- [ ] `inviteToken` stored in `sessionStorage` and consumed by `setup/page.tsx`
- [ ] Error state on invite page renders when sessionId or inviteToken is missing

## VR-34 — Audio Performance Under GPU Load

### Repro Steps

**Scenario A — RNNoise ON: native NS disabled**
1. Join session with RNNoise enabled. Check DevTools → MediaStream.
2. Expected: `noiseSuppression: false` in active audio track constraints.

**Scenario B — RNNoise OFF: native NS enabled**
1. Repeat with RNNoise disabled.
2. Expected: `noiseSuppression: true` in track constraints.

**Scenario C — Toggle RNNoise mid-session**
1. Join with RNNoise off. Enable via settings.
2. Expected: `applyConstraints({ noiseSuppression: false })` called on track.
3. Disable RNNoise. Expected: native NS re-enabled.

**Scenario D — RNNoise fails to load**
1. Block `/audio-worklets/rnnoise.wasm` in DevTools Network. Enable RNNoise.
2. Expected: Fallback branch fires, native NS remains active.

**Scenario E — SDP munging verification**
1. Join 2-user session. Inspect SDP via `chrome://webrtc-internals`.
2. Expected: `maxaveragebitrate=32000;useinbandfec=1;cbr=1` in Opus fmtp line, `a=ptime:40`.

**Scenario F — Subjective quality under load**
1. Run GPU benchmark + 2-user voice session with RNNoise.
2. Expected: Audio intelligible, not robotic.

**Scenario G — reconnectNoiseSuppressionOnExistingPipeline**
1. Enable RNNoise → disable → re-enable multiple times.
2. Expected: Audio pipeline remains functional throughout.

### Edge Cases

- **`optimizeAudioSenderParams` never called:** Dead code. SDP munging alone handles bitrate.
- **SDP has no fmtp line for Opus:** Handled by `else` branch inserting fresh fmtp line. Test on Firefox.
- **SDP has no Opus rtpmap:** `opusMatch` is `null`, returns original SDP unmodified.
- **`a=ptime:` already present:** Regex replaces it with 40.
- **`track.applyConstraints` not supported:** Caught silently. Verify Safari doesn't throw.
- **CBR and packet loss:** CBR sends constant rate even for silence. FEC compensates on lossy networks.

### Regression Checklist

- [ ] Voice connects between 2 users after SDP changes
- [ ] Voice connects with 3+ users (mesh)
- [ ] Audio audible in Chrome ↔ Chrome call
- [ ] Audio audible in Chrome ↔ Firefox call
- [ ] Mute/unmute still works
- [ ] Channel switch tears down and re-establishes voice correctly
- [ ] RNNoise toggle doesn't drop the call
- [ ] RNNoise can be toggled multiple times without pipeline corruption
- [ ] Screen share audio not affected by Opus SDP munging
- [ ] Leave + rejoin doesn't leave ghost audio nodes
- [ ] `app-config.json` is valid JSON; TypeScript build succeeds

**Scenario C — Room code input (join form)**

2. Expected: No autofill suggestions.

**Scenario D — Max users field**
1. Select Private room type. Click max users input.

**Scenario F — Keyboard and form submission**

2. Expected: Form advances normally. `onKeyDown` handler fires correctly.

### Edge Cases

- **Firefox autofill:** `autoComplete="one-time-code"` may behave differently. Verify no autofill.

- **Label association:** Click `<Label>` → verify input gets focused (a11y check for `display-alias`, `room-password`, `room-secret`).
- **Chrome learning:** After typing and submitting a name, Chrome may start suggesting it. Verify `one-time-code` continues to suppress.
- **Tauri WebView2 autofill:** Windows WebView2 has its own autofill. Test in desktop app.
- **Bitwarden:** May still inject icons (out of scope but worth noting).
