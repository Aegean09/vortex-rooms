## VR-36: Android Tauri app -- audio cuts out after screen lock

### Files Changed
- `src/lib/webrtc/hooks/use-mobile-background-recovery.ts` -- added auto-mute on background, AudioContext keep-alive
- `src/lib/webrtc/provider.tsx` -- added `onBackgroundAutoMute` callback, Firestore update
- `src-tauri/Cargo.toml` -- conditional deps for Android (no single-instance, no updater)
- `src-tauri/tauri.conf.json` -- Android icons, mobile deep-link config
- `src-tauri/tauri.android.conf.json` -- Android-specific build config (minSdkVersion 24)
- `src-tauri/android-permissions.xml` -- reference manifest for WAKE_LOCK, FOREGROUND_SERVICE
- `src-tauri/capabilities/default.json` -- description update only

### Acceptance Criteria Review

| AC | Status | Notes |
|----|--------|-------|
| AC-1: Incoming audio continues 5+ min | PARTIAL | AudioContext keep-alive dispatches `vortex:resume-audio` every 5s. Provider listener resumes suspended contexts. However, the keep-alive only calls `resume()` on AudioContext -- it does NOT acquire a Wake Lock or Foreground Service at the Rust/OS level. On Android, the WebView process itself may be killed by the OS after extended background time, and `setInterval` may not fire reliably in a backgrounded WebView. The `android-permissions.xml` declares `WAKE_LOCK` and `FOREGROUND_SERVICE_MEDIA_PLAYBACK` but no Rust code or Tauri plugin actually acquires them. |
| AC-2: Auto-mute on screen lock | PASS | `autoMuteOutgoingAudio` disables tracks on both `localStream` and `rawStream`, then calls `onBackgroundAutoMute` which sets `isMuted: true` in state and Firestore. |
| AC-3: User sees muted on return | PASS | `setIsMuted(true)` in provider ensures UI state reflects muted. User must manually unmute. |
| AC-4: AudioContext resumed on foreground | PASS | `resumeAudioContexts` called on visibility change to visible (existing behaviour preserved). Keep-alive stops on foreground return. |
| AC-5: Mic reconnected if tracks ended | PASS | Existing `handleTrackEnded` logic preserved unchanged. |
| AC-6: Desktop unchanged | PASS | `isMobileDevice()` gates auto-mute and keep-alive behind UA check. Desktop/browser unaffected. |

### Issues Found

**[VR-36-BUG-1] MEDIUM: Wake Lock / Foreground Service not actually acquired**
The `android-permissions.xml` file declares `WAKE_LOCK` and `FOREGROUND_SERVICE_MEDIA_PLAYBACK` permissions, but no code acquires a wake lock or starts a foreground service. The permissions alone do nothing -- they need to be actively used. The `setInterval` keep-alive in a backgrounded Android WebView may not fire reliably. This means AC-1 may fail in practice on some devices/OEMs.

**Mitigation:** The current implementation is a reasonable best-effort approach. The `android-permissions.xml` is documented as a reference file to merge manually. A Tauri plugin for wake lock or the Web Wake Lock API (`navigator.wakeLock.request('screen')`) would strengthen this, but that could be a follow-up.

**[VR-36-BUG-2] LOW: `useCallback` dependency on `localStream` / `rawStream` in `autoMuteOutgoingAudio`**
`autoMuteOutgoingAudio` captures `localStream` and `rawStream` via closure. If either stream is replaced (e.g., device switch) between the callback being created and the visibility change firing, the callback will reference stale streams. The effect re-registers on dependency changes (`autoMuteOutgoingAudio` is in the dependency array), so in practice this should be fine, but the event listener is removed and re-added on every stream change, which is a minor concern.

**[VR-36-OBSERVATION] Cargo.toml feature flag `desktop`**
The `[features]` section adds `default = ["desktop"]` and `desktop = ["tauri/tray-icon"]`. This is correct for Tauri 2 -- tray icons are desktop-only. The Android build would use `--no-default-features`. Verified that `tauri-plugin-single-instance` and `tauri-plugin-updater` are correctly behind `[target.'cfg(not(target_os = "android"))'.dependencies]`.

## VR-37: Tauri desktop app double-redirect on deep-link

### Files Changed
- `src-tauri/src/lib.rs` -- refactored `parse_deep_link_path` to `parse_deep_link_route`, `focus_and_navigate` now stores invite token in sessionStorage and navigates directly to `/session/{id}/setup`
- `src-tauri/tauri.conf.json` -- added mobile deep-link host configuration

### Acceptance Criteria Review

| AC | Status | Notes |
|----|--------|-------|
| AC-1: No double redirect | PASS | Rust now navigates directly to `/session/{id}/setup` and stores the invite token in sessionStorage, bypassing the invite page entirely. This eliminates the redirect loop. |
| AC-2: Existing window focused | PASS | `focus_and_navigate` calls `unminimize`, `show`, `set_focus`. Single-instance plugin passes `argv` with deep-link URL. |
| AC-3: Cold start navigation | PASS | `get_current()` handles cold start URLs, deferred by 500ms via `std::thread::spawn` + `sleep` to wait for WebView readiness. |
| AC-4: Web flow unchanged | PASS | Invite page still has `tryDesktopDeepLink` for browser-to-desktop handoff. `isTauri()` check prevents re-triggering inside Tauri. |
| AC-5: No stuck spinner | PASS | If deep-link succeeds, page shows "Opening Vortex...". If not, falls back to web flow. |

### Issues Found

**[VR-37-BUG-1] MEDIUM: sessionStorage key consistency -- Rust vs TypeScript**
The Rust code uses:
```rust
"sessionStorage.setItem('vortex-invite-token-{}', '{}'); window.location.href = 'https://vortex-rooms.com/session/{}/setup';"
```
The TypeScript invite page uses:
```typescript
sessionStorage.setItem(`vortex-invite-token-${sessionId}`, inviteToken);
```
The setup page reads:
```typescript
sessionStorage.getItem(`vortex-invite-token-${sessionId}`)
```
**Key format matches.** The Rust code interpolates `session_id` directly into the key pattern `vortex-invite-token-{session_id}`, which produces the same key as the TypeScript template literal. **No mismatch.**

**[VR-37-BUG-2] LOW: XSS via session ID / invite token in `eval()`**
The Rust `focus_and_navigate` function interpolates `session_id` and `invite_token` into a JavaScript string passed to `window.eval()`. The code does `replace('\'', "\\'")` to escape single quotes. However, this is insufficient for full XSS prevention -- a session ID or invite token containing backslashes, newlines, or other special JS characters could break out of the string.

**Actual risk is LOW** because:
1. `parse_deep_link_route` validates the URL structure (4 segments, specific positions).
2. Session IDs are alphanumeric codes (12 chars, generated server-side).
3. Invite tokens are also server-generated UUIDs.
4. The deep link comes from the OS, not arbitrary user input in a web context.

**Recommendation:** For defense-in-depth, consider using `serde_json::to_string()` to properly JSON-escape the values, or validate that they contain only `[a-zA-Z0-9\-_]` before interpolation.

**[VR-37-BUG-3] LOW: Cold start 500ms sleep is fragile**
The cold start handler uses `std::thread::sleep(Duration::from_millis(500))` to wait for the WebView. On slow devices (particularly Android), the WebView may not be ready in 500ms. There is no retry or readiness check. If `eval()` fires before the WebView has loaded, the navigation will silently fail.

**Recommendation:** Consider using Tauri's `on_page_load` or `WebviewWindow::on_navigation` event to trigger navigation once the WebView is actually ready.

## VR-38: Typing indicators in chat

### Files Changed
- `src/app/session/[sessionId]/hooks/use-typing-indicator.ts` -- new hook (debounced Firestore writes, stale expiry)
- `src/app/session/[sessionId]/hooks/index.ts` -- export new hook
- `src/app/session/[sessionId]/page.tsx` -- wire up typing indicator to ChatArea
- `src/components/chat-area/chat-area.tsx` -- render typing indicator UI, wire input onChange
- `src/interfaces/session.ts` -- added `typingInChannel` and `typingAt` to User interface
- `tailwind.config.ts` -- added `typing-dot` keyframe animation

### Acceptance Criteria Review

| AC | Status | Notes |
|----|--------|-------|
| AC-1: Single user typing text | PASS | `formatTypingText` returns `"{name} is typing..."` |
| AC-2: Two users typing | PASS | `"{name1} and {name2} are typing..."` |
| AC-3: Three+ users typing | PASS | `"{name1}, {name2}, and {count} others are typing..."` (correct pluralization for `other` vs `others`) |
| AC-4: Appears within 1s | PASS | Debounce is 300ms. Firestore real-time listener delivers within ~200-500ms. Total < 1s. |
| AC-5: Disappears within 3s of stop | PARTIAL | Expiry is 5000ms (`TYPING_EXPIRY_MS`), stale check interval is 2000ms. Worst case: user stops typing, then up to 5s + 2s = 7s before indicator disappears. Spec says 3s. |
| AC-6: Disappears immediately on send | PASS | `onMessageSent` clears debounce timer and writes `typingInChannel: null` to Firestore. Real-time listener will propagate removal. |
| AC-7: Self not shown | PASS | Filter excludes `u.id === authUser.uid`. |
| AC-8: Works in E2E rooms | PASS | Typing state uses `typingInChannel` field on user doc (not encrypted). Uses already-decrypted `u.name` for display. |
| AC-9: Ephemeral state | PASS | Stored on user doc which is deleted on leave. Cleanup effect clears debounce timer. |
| AC-10: Subtle animation | PASS | Three dots with `animate-typing-dot` keyframe (scale + opacity pulse, 1.4s cycle, staggered 0.2s delays). Consistent with dark-mode design. |
| AC-11: Scoped to active text channel | PASS | Filter checks `u.typingInChannel !== activeTextChannelId`. Channel change clears typing state. |

### Issues Found

**[VR-38-BUG-1] MEDIUM: Typing indicator disappears too slowly (spec says 3s, implementation is up to 7s)**
`TYPING_EXPIRY_MS` is 5000ms and `STALE_CHECK_INTERVAL_MS` is 2000ms. The spec (AC-5) says the indicator should disappear within 3 seconds of the user stopping typing. Worst case is 5s + 2s = 7s.

**Fix:** Reduce `TYPING_EXPIRY_MS` to 3000ms and `STALE_CHECK_INTERVAL_MS` to 1000ms. This would give a worst case of 4s, which is closer to spec. Alternatively, the debounce could re-fire the Firestore write periodically while typing to keep `typingAt` fresh, and use a shorter expiry.

**[VR-38-BUG-2] MEDIUM: No cleanup of typing state on unmount / page navigation**
The unmount cleanup only clears the debounce timer but does NOT write `typingInChannel: null` to Firestore. If a user navigates away from the session page (e.g., back to join page) without explicitly sending a message, their typing indicator could persist for other users until the user doc is deleted by presence cleanup (which may take up to 60s depending on the stale user cleanup job).

**Fix:** Add a Firestore write in the cleanup effect:
```typescript
useEffect(() => {
  return () => {
    if (debounceTimerRef.current) {
      clearTimeout(debounceTimerRef.current);
    }
    // Clear typing state on unmount
    writeTypingState(null);
  };
}, [writeTypingState]);
```

**[VR-38-BUG-3] LOW: `onInputChange` only fires when input has non-whitespace content**
In `chat-area.tsx`:
```tsx
onChange={(e) => {
  setNewMessage(e.target.value);
  if (e.target.value.trim()) onInputChange?.();
}}
```
This means if the user types spaces or deletes all text, `onInputChange` is not called. The typing indicator will persist until expiry even though the input is effectively empty. This is a minor UX issue -- the indicator should clear when the input is emptied.

**Fix:** Call `onMessageSent()` (or a new `onInputCleared`) when the input becomes empty.

**[VR-38-BUG-4] LOW: Encrypted user names may show raw encrypted string in typing indicator**
The typing indicator displays `u.name`. In E2E rooms where `USERNAME_DECRPYTION_ENABLED` is false (currently the case per `app-config.json`), or where the metadataKey hasn't been received yet, `u.name` may be an encrypted ciphertext string rather than a readable name. The spec suggests showing "Someone is typing..." as fallback.

**Current risk:** The `USERNAME_DECRPYTION_ENABLED` flag is `false` in config, so names should always be plaintext on the user doc. The encrypted name is in `encryptedName` field, not `name`. So this may not be an actual issue in practice -- but worth verifying.

**[VR-38-PASS] Firestore rules**
The user document update rule (line 117-120 of `firestore.rules`) does NOT use `keys().hasOnly()` -- it only validates `sessionId`, `id`, and username length. Therefore `typingInChannel` and `typingAt` can be written without rule changes. No rule update needed.

**Note:** The lack of `hasOnly()` on user doc updates means any arbitrary field could be injected. This is a pre-existing concern, not introduced by VR-38.

**[VR-38-PASS] Animation CSS**
The `typing-dot` keyframe in `tailwind.config.ts` is correctly defined:
- `0%, 80%, 100%`: opacity 0.3, scale 0.8
- `40%`: opacity 1, scale 1
- Animation duration: 1.4s infinite
- Three dots staggered at 0, 0.2s, 0.4s delays via inline `style`

This produces a smooth pulsing dots effect. The `aria-hidden="true"` on the dots container is correct since the text already conveys meaning.
